# Architecture

## Overview

The Utility Bills app is a config-driven data pipeline with four Docker containers that share a single volume. The central idea is that all vendor-specific logic (how to find emails, how to parse PDFs, what regex to use) lives in `vendors.yaml` rather than in code. This means adding or changing a utility provider requires only a config edit.

## Shared Volume Layout

All four containers mount the same NAS directory (`/volume1/bills`) as `/data`:

```
/data/
├── config/
│   └── vendors.yaml          ← master vendor config, read by scraper + parser
├── credentials/
│   ├── gmail_credentials.json  ← OAuth2 client config (from Google Cloud)
│   └── gmail_token.json        ← OAuth2 refresh token (generated at first run)
├── raw/                        ← PDF bills, renamed to canonical format
│   └── YYYY-MM_vendor_amount_EUR.pdf
└── dashboard.html              ← generated by parser, served by api
```

> `credentials/` and `raw/` are excluded from version control (see `.gitignore`).

## Container Details

### bills_gmail_scraper

**Role:** Polls Gmail daily, downloads PDFs, triggers parsing, sends Telegram summary.

- **Runtime:** Python 3.12, `schedule` library
- **Schedule:** Daily at `POLL_DAILY_TIME` (default `09:00`)
- **Auth:** Gmail OAuth2, scope `gmail.modify`
- **Key source files:** `main.py`, `vendor_config.py`

**Processing flow:**
1. Load `vendors.yaml` and build a Gmail search query from all `sender_domains` + global `subject_keywords`
2. Fetch unread matching emails not already in `raw_emails` table
3. For each email:
   - Download PDF attachment → save to `/data/raw/` with original filename
   - `POST /parse` to `bills_parser:8001` with the file path
   - **No-PDF variant** (Telia Internet): extract amount, date, invoice number directly from email HTML body
   - **Label-based search** (House Insurance): search by Gmail label instead of sender
4. Apply `skip_rules` from vendor config (e.g. skip pellet *offers* and *prepayments*)
5. Label processed emails `UTILITY_BILL`, mark as read
6. Record message IDs in `raw_emails` to prevent reprocessing
7. Send a Telegram batch summary of all newly processed bills
8. `POST /generate-dashboard` to regenerate `dashboard.html`

---

### bills_parser

**Role:** Receives PDF paths, extracts structured data via regex, writes to DB, generates dashboard.

- **Runtime:** Python 3.12, FastAPI on uvicorn `:8001`
- **Key source files:** `vendor_config.py`, `parser_engine.py`, `dashboard_generator.py`, `db.py`

**Endpoints:**

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/parse` | Parse a single PDF, write to DB, rename to canonical filename |
| `POST` | `/generate-dashboard` | Regenerate `/data/dashboard.html` from DB |

**Parse flow:**
1. Extract raw text from PDF using MuPDF (PyMuPDF / `fitz`)
2. Classify the vendor using `vendor_config.classify()`:
   - Match filename against `filename_slugs`
   - Fallback: match PDF text against `text_keywords`
   - Apply `require_keywords` and `exclude_keywords` guards
3. Extract total amount using the vendor's `total_patterns` (first matching regex wins)
4. Optionally extract consumption (`energy_kwh`, `gas_m3`, `water_m3`, `other_units`)
5. Optionally extract billing period dates
6. Apply `special` flags:
   - `pro_rate_across_period`: split insurance cost across billing months
   - `spread_heating_season`: spread pellet cost across Sep–Apr
   - `month_alignment`: align service month to `billing_period_start` or `billing_period_end`
7. Rename PDF to `YYYY-MM_vendor_amount_EUR.pdf`
8. Write row to `parsed_bills` table

### bills_api

**Role:** Public-facing HTTP server — serves the dashboard, PDF files, and a JSON REST API.

- **Runtime:** Python 3.12, FastAPI on uvicorn `:8000`, exposed externally as `:8888`

**Endpoints:**

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/` | — | Serve `dashboard.html` |
| `GET` | `/bills/{filename}` | — | Serve a raw PDF from `/data/raw/` |
| `GET` | `/health` | — | Health check |
| `GET` | `/api/bills` | API key | List bills with filters (vendor, date range, pagination) |
| `GET` | `/api/bills/monthly-totals` | API key | Aggregated monthly totals per vendor |
| `GET` | `/api/bills/trends` | API key | Year-over-year comparisons |
| `GET` | `/api/bills/per-unit-costs` | API key | Unit cost analytics (€/kWh, €/m³) |
| `GET` | `/api/meter-readings` | API key | List manual meter readings |
| `POST` | `/api/meter-readings` | API key | Add a manual meter reading |
| `POST` | `/api/bills/manual` | API key | Add a bill entry manually |

Authentication uses the `X-API-Key` HTTP header.

### bills_mariadb

**Role:** Persistent data store.

- **Database:** `utility_bills`

**Tables:**

| Table | Description |
|-------|-------------|
| `parsed_bills` | One row per bill: vendor, dates, amounts, consumption, file path, parse status |
| `raw_emails` | Gmail message IDs that have been processed (deduplication) |
| `meter_readings` | Manual meter entries (date, vendor, reading value) |
| `parsing_errors` | Failed parse attempts with error messages |

**`parsed_bills` columns:**

```sql
id                  INT AUTO_INCREMENT PRIMARY KEY
vendor_category     VARCHAR(64)     -- e.g. "electricity", "gas"
invoice_date        DATE
billing_period_start DATE
billing_period_end  DATE
total_amount        DECIMAL(10,2)
energy_kwh          DECIMAL(10,3)   -- electricity or gas in kWh
gas_m3              DECIMAL(10,3)
water_m3            DECIMAL(10,3)
other_units         DECIMAL(10,3)   -- e.g. pellets in kg
unit_type           VARCHAR(16)     -- "kg", "kWh", etc.
raw_pdf_path        VARCHAR(512)    -- path or "email:<id>" for no-PDF variants
status              VARCHAR(16)     -- "success" or "error"
created_at          DATETIME
```

## Dashboard Generator

`dashboard_generator.py` (tracked in this repo as `dashboard_generator_patched.py`) queries the DB and emits a single self-contained HTML file with all data embedded as JavaScript constants.

**Data flow:**

```
DB (parsed_bills)
    │
    ▼  _query_all()
Raw rows
    │
    ▼  generate_dashboard()
Aggregated Python dicts
    • bill_data[vendor][YYYY-MM] = EUR
    • elec_kwh[YYYY-MM] = kWh
    • gas_m3, gas_kwh, water_m3, water_eur_svc
    • elec_net_svc, gas_net_svc  (network cost by service month)
    • pellet_by_season[year]     (Sep-Apr grouping)
    │
    ▼  _TEMPLATE_TOP + JS constants + _TEMPLATE_BOTTOM
dashboard.html
```

**Special aggregation rules:**

| Vendor | Rule |
|--------|------|
| Pellets | Invoices grouped into heating seasons (Sep–Apr). Season total divided evenly across 8 months. |
| House Insurance | Annual/quarterly invoice split proportionally by day count across covered months. |
| Water | Service month taken from `billing_period_end` (meter reading period lags invoice date). |
| Electricity Network / Gas Network | Aligned to `billing_period_start` (one month behind invoice). |

## Config-Driven Vendor System

The key design principle: `vendors.yaml` is the single source of truth for everything vendor-specific. Both the scraper and parser load this file at startup via `vendor_config.py`.

```
vendors.yaml
    │
    ├─► scraper reads:  gmail.sender_domains, gmail.label, special.no_pdf_query
    │                   special.skip_rules
    │
    └─► parser reads:   classification.filename_slugs / text_keywords
                        parsing.total_patterns
                        parsing.consumption.patterns
                        parsing.billing_period.patterns
                        special.*
```

See [`VENDOR_CONFIG.md`](VENDOR_CONFIG.md) for the full field reference.

    ▼

    │

    bills_gmail_scraper → bills_parser:8001
    bills_parser        → bills_mariadb:3306
    bills_api           → bills_mariadb:3306

No container other than `bills_api` is reachable from outside the Docker bridge.

## Operational Notes

- **Code changes** require rebuilding the Docker image or editing files inside the running container and copying back to the source directory. Changes inside a container are lost on recreate.
- **Config changes** (`vendors.yaml`) only require restarting the scraper and parser containers — no rebuild needed.
- **Module caching:** uvicorn keeps Python modules in memory. Always `docker restart bills_parser` after any `.py` change.
- **Gmail token expiry:** The OAuth refresh token never expires if the Google Cloud app is published to *Production*. In *Testing* mode tokens expire after 7 days. Use `refresh_gmail_token.py` to regenerate interactively.
- **Watchtower:** If running Watchtower for auto-updates, ensure your source directory on the NAS stays in sync with what is baked into the Docker image.
